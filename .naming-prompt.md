You are a senior naming consultant who specializes in open-source developer tools. I need you to help me find a great project name for a piece of software I'm about to open-source on GitHub.

================================================================
PART 1 — THE PROJECT
================================================================

I'm building a **virtual teleprompter** desktop app. It is built with **Electron** and runs on Linux, macOS, and Windows.

What it does, in concrete terms:

1. A small "control" window lets the user paste a script, then adjust:
   - scroll speed
   - text size, line height, letter spacing, side margin
   - text color and a highlight color
   - background opacity (so the user can see what they're working on through the prompter)
   - background dim (contrast — keep the backdrop readable without losing see-through)
   - mirror mode (for use with a physical beam-splitter glass)
   - bold, uppercase, optional text outline
   - reading-line on/off
   - window width, height, and a position preset (top-center, top-left, top-right, bottom-center)

2. A separate "prompter" window opens as a **transparent, frameless, always-on-top overlay**. It scrolls the script from bottom to top across a fixed horizontal strip. The user can:
   - resize and drag the window
   - pause / resume / reset the scroll
   - nudge speed up/down in 5 px/s steps
   - toggle a **true OS-level click-through mode** (the main process calls `setIgnoreMouseEvents(true, { forward: true })`), so all clicks and wheel events pass through to whatever app is behind, while the prompter keeps rendering on top. The HUD's interactive children (pause, reset, lock, close) stay reachable because of the `forward: true` flag.

3. Settings persist to a JSON file in the Electron `userData` directory. A live IPC bridge keeps the two windows in sync, so editing the script or any slider in the control window updates the prompter immediately.

The user is typically a video creator, public speaker, lecturer, podcaster, or anyone recording themselves who needs the text scrolling on screen while their camera records. The "see through to your actual work" feature is the killer one — they can keep editing, coding, presenting slides, etc., behind the prompter.

================================================================
PART 2 — NAMING STYLE I WANT
================================================================

My **original** working name was **`scrdr`**, which I held as a contraction of "screen reader" (i.e. the thing that displays readable text on the screen). The vibe I liked:

- **5–6 letters** (short, like `scrdr`)
- **One or more vowels dropped** to make it feel like a quick keystroke
- **Easy to type, easy to say, easy to remember after hearing it once**
- **Looks like a real word or could plausibly be one** — not random consonants
- **On-theme** — evokes reading text, scrolling text, speaking, presenting, or seeing through something

What I did **NOT** want:
- A descriptive plain word like `prompt`, `script`, `reader`, `teleprompter`, `prompter`. (Too generic — the JS ecosystem is full of these, npm is hostile to them, and they don't read as a brand.)
- An `Open*` prefix (OpenSomething). I tried that direction; everything in the obvious `open*` space is taken on both GitHub and `.com`.
- A cutesy, abstract, or non-English word. I'm fine with a coined word as long as it sounds English-ish and the meaning is recoverable.

I am explicitly **open to your own ideas and directions** as long as the constraints above are met. Surprise me.

================================================================
PART 4 — HARD CONSTRAINTS
================================================================

For every name you propose, you must attempt to verify:

1. **GitHub user/org namespace** — fetch `https://github.com/<name-lowercased>` and confirm it returns 404. (Bonus: also check that the org is not a parked squat with zero public repos, because some squatters squat and never release.)
2. **`.com` availability** — use a WHOIS lookup (e.g. `https://www.whois.com/whois/<name>.com`). Parked domains on Squadhelp, HugeDomains, etc. count as **taken** (they're being held for sale, often at 4-figure prices, and they are deliberate brand-protection buys).
3. **Brand collision in Google / DuckDuckGo** — search `"<name>"` and `"<name>" teleprompter` and `"<name>" app`. If a popular, well-known product or company already uses the name (especially in a domain that overlaps with mine: developer tools, productivity, content creation, video, audio, reading, writing, presentations), it's **not safe** even if GitHub and `.com` are technically free.
4. **Legibility / pronounceability** — a colleague hearing it once should be able to type it from memory and pronounce it without being told.
5. **Meaning recoverable** — within two seconds of seeing the name, the user should be able to guess roughly what the app does. This is the "looks like a real word" property.

If you cannot fully verify a name because of a bot challenge on a search engine, say so explicitly in your output and explain what you did check.

================================================================
PART 5 — STRATEGIC GUIDANCE (this is the part where I want you to think hard)
================================================================

- The reason my first 18-or-so attempts all failed is that I kept reaching for **real English words** (`readr`, `scrollr`, `screed`, `liner`, `scread`, …) and the obvious real-word space is largely claimed by 2026. So: **lean into coinage**. The best name is probably a word that doesn't quite exist yet but is *almost* a word.
- Look at how other successful developer tools are named: `npm`, `vite`, `pnpm`, `brew`, `tmux`, `fzf`, `bat`, `ripgrep`, `lazygit`, `ddev`, `mise`. Most are **3–6 letters, all-lowercase, monomorphemic, easy to spell**. That's the target register.
- Consider word-formation techniques: clipping (`script` → `scrip`), blending (`screen + read` → `screed`), vowel-deletion (`reader` → `reedr` — already taken, but the *technique* is sound), affixation (`promp + tly`), back-formation, or genuine coinage.
- Consider semantic fields I should mine:
  - **Reading**: page, line, verse, prose, voice, cue, prompt, line, word, text, script, scroll, roll, drift, flow, stream, ribbon
  - **Speaking / presenting**: cue, say, speak, tell, talk, voice, line, prompt, roll, stage, on-air
  - **Visibility / transparency**: glass, veil, scrim, lens, clear, see-through, light, beam
  - **The action of scrolling / rolling**: roll, scroll, drift, run, flow, stream, queue, ticker, line
  - **The prompter object itself**: glass, beam, hood, rig, monitor, display, stage
- A name that **sounds like a verb** is a plus (e.g. `fzf` doesn't, but `bat` does, `mise` does).
- A name where the letters hint at the meaning (e.g. `scrn` → screen, `rdr` → reader) is a plus — that's why `scrdr` worked for me in the first place.
- Don't shy away from names that are slightly awkward to type at first, as long as they're memorable. `tmux` is awkward and won.
